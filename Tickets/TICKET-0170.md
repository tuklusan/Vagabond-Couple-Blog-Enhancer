# TICKET-0170: [vision_client.py] 1J cannot fetch non-Blogger images (Wikimedia 403 + no downscale path)
Status: Closed
Priority: Medium
Type: Bug
Created: 2026-07-04
Description: Observed live during the first full-post 1J audit (run alaska2v1, 424 images): image 19 is a Wikimedia Commons original (https://upload.wikimedia.org/wikipedia/commons/d/d5/MS_Statendam%28js%2902.jpg) and failed with "403 Client Error: Forbidden" -- the only fetch failure of the run (423/424 audited). Two stacked causes: (1) Wikimedia rejects requests bearing the default python-requests User-Agent (their UA policy requires an identifying agent string); (2) even with the 403 fixed, the URL has no Blogger size token, so the downscale rewrite is a no-op and the 5.6MB original blows the 160KB inline-b64 cap.
Steps to Reproduce: vision_client.fetch_image("https://upload.wikimedia.org/wikipedia/commons/d/d5/MS_Statendam%28js%2902.jpg") before the fix -> (None, "fetch failed: 403 ...").
Notes: Fixed both layers, each verified live against the exact failing URL: (1) FETCH_HEADERS with an honest identifying User-Agent (tool name + repo URL, per Wikimedia's UA policy) sent on every image fetch; (2) downscaled_url() now recognizes Wikimedia Commons originals and rewrites them to the thumbnail path scheme (.../thumb/<x>/<xy>/<File>/<N>px-<File>). Wikimedia additionally rejects arbitrary thumbnail widths from unauthenticated clients with 400 "Use thumbnail sizes listed..." (also observed live at 512px), so the rewrite uses the standard widths 500px (primary) / 250px (retry). Result: the failing image now fetches as a 49KB 500px JPEG. Other third-party hosts remain best-effort: no size-rewrite is attempted, the 160KB cap and per-image failure counters still protect the audit, and a failed image is a logged skip, never a halt. New tests: rewrite_wikimedia_thumb, rewrite_wikimedia_retry_250px, ua_header_present. NOTE: the alaska2v1 run that surfaced this was already past 1J when the fix landed (module loaded in memory), so its artifact legitimately shows fetch_fail=1; the progress cache will pick image 19 up on the next run of that post.
