# TICKET-0132: Add operator-supplied prior/next-post URLs for genuine, linked lead-in/lead-out
Status: closed
Priority: High
Type: Improvement
Created: 2026-07-03
Description: The human workflow is fed the current/prior/next post's live URLs and is required to scan the prior/next posts to write a genuine lead-in (referencing+linking the prior post) and lead-out (referencing+linking the next post) -- e.g. POSTED's Arsenalna intro says 'As promised in our Kyiv, Ukraine post...' and its closing links to the Georgia post. The orchestrator had NO equivalent mechanism at all -- confirmed as the root cause of an ungrounded, disconnected 'poetic' closing paragraph the reviewer flagged during the arsenalna1 diff cycle (step10_journey_significance had nothing concrete to anchor its closing thought to). IMPLEMENTED: (1) new --current-url/--prior-url/--next-url CLI args (orchestrator/__main__.py), stored in sctx.context before the sequence runs; (2) context_extractor.fetch_post_gist(url) -- best-effort fetch of a live post's real title + gist (meta description or first <p>), returns None on ANY failure (network down, 404, timeout) rather than raising -- lead-in/lead-out is opt-in and never blocks a run; (3) new deterministic sequencer node lead_context_node() (runs right after context_extraction) populates context['prior_post']/['next_post'] when the corresponding URL was supplied; (4) step6_first_body_paragraph's writer/reviewer now build a lead-in from the REAL fetched prior_post (title/url/gist) when present, and step10_journey_significance's writer/reviewer build a lead-out from the REAL fetched next_post, both explicitly forbidden from inventing what the referenced post covers; (5) added _with_lead_link_check() wrapper -- a deterministic guard requiring the real prior_post/next_post URL to literally appear as a link in the output whenever one was supplied, so the writer can't silently drop a real lead-in/lead-out link. Verified: full deterministic test suite green including the dry full-sequence walk (lead_context_node no-ops cleanly when no URLs are supplied, backward compatible). NOT yet verified against a LIVE network fetch of the real Arsenalna/Kyiv/Georgia URLs -- next step.
