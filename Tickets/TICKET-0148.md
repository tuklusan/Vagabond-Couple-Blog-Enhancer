# TICKET-0148: [reviewer_client.py] Pause-turn continuation logic is broken, breaking web-groun
Status: closed
Priority: High
Type: Bug
Created: 2026-07-03
Description: In `_run_conversation`, when `stop_reason == "pause_turn"` (e.g., during web search), the code appends only the assistant response to `messages` but omits the required subsequent user message. The next API call will then have a messages array ending with an assistant role, which violates the Anthropic message alternation rule and will cause a `BadRequestError`. This error is caught in `_review_with_claude`, leading to an immediate fallback to DeepSeek (or the next web_search tool variant), effectively losing the ability to complete a multi-turn Claude conversation with live web access. As a result, fact-checks that need a second turn will never be certified by Claude, undermining the reviewer's trust-anchor guarantee. | Suggestion: After appending the assistant message, insert a new user message (e.g., {"role": "user", "content": "Continue."}) before continuing the loop, to maintain correct role alternation. | File: orchestrator/reviewer_client.py | Severity: critical
Steps to Reproduce: 
Notes: FALSE POSITIVE, no action: the current code (append the assistant's paused-turn content to messages, then resend the SAME messages array with no new user turn) is Anthropic's documented pause_turn continuation pattern -- confirmed against the claude-api skill's manual-agentic-loop reference: 'Handle pause_turn in your agentic loop... re-send the user message and assistant response... Do NOT add an extra user message like Continue. -- the API detects the trailing server_tool_use block and knows to resume automatically.' Inserting a synthetic user message as the ticket suggested would NOT fix a real bug -- there wasn't one. This mirrors an earlier confirmed false positive in this project (TICKET-0072, same pause_turn append-assistant pattern).
