# TICKET-0169: [image_audit.py] Audit progress cached only by image URL, ignoring text content
Status: Closed
Priority: High
Type: Bug
Created: 2026-07-04
Description: The `1J_image_audit_progress` artifact keys verdicts by `src` URL alone. If an image’s alt text, title, or caption is updated after a previous audit, a re‑run reuses the stale verdict and may miss new contradictions or fail to apply corrections. This defeats the purpose of re‑auditing when post content changes. | Suggestion: Include a hash of the relevant text fields (alt, title, caption) in the cache key, or store a digest alongside the verdict and re‑audit if the current content hash differs from the stored one. | File: orchestrator/image_audit.py | Severity: critical
Steps to Reproduce: 
Notes: VALID (low likelihood in current usage, but a real staleness class), FIXED per the reviewer's second suggested form: each cached verdict now stores `text_key` -- a sha256 digest (16 hex chars) of the alt/title/caption it was actually judged against (`_text_key(rec)`), and `audit_images` only reuses a cache entry when the digest matches the CURRENT record's text; any mismatch (operator edit, previously applied correction, changed caption) forces a re-audit and the refreshed entry carries the new digest. Likelihood note: in current usage a resumed run re-audits the same untouched source (1J runs before any Phase 5 mutation, and fresh run_ids get fresh artifact dirs), so the stale path needed an operator hand-edit between resume attempts -- but the guard is one hash comparison and removes the whole class. New test: test_progress_cache_reuse_and_staleness (verifies cache hit = zero VLM calls, stale key = re-audit, refreshed entry carries the current digest). Image-audit suite 34 checks green.
