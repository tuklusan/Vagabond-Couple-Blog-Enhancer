# TICKET-0184: [_review_gate.py] Fingerprint file path mismatch prevents false-positive suppres
Status: Closed
Priority: High
Type: Bug
Created: 2026-07-04
Description: `load_dismissed_fingerprints` stores the full relative file path (e.g., `src/main.py`) from ticket titles, but `_is_dismissed` compares only the basename (`main.py`) against it. This causes the Jaccard overlap to never reach the threshold for any non-root file, so dismissed findings are never recognised. As a result, critical findings previously triaged as false positives will repeatedly block pushes and generate duplicate tickets, undermining the suppression mechanism. | Suggestion: Change `_is_dismissed` to compare the full relative path instead of just the basename: e.g., replace `base = Path(str(file_path)).name.lower()` with `base = Path(str(file_path)).as_posix().lower()` or keep both fields in the fingerprint and match on the original path. | File: .githooks/_review_gate.py | Severity: critical
Steps to Reproduce: 
Notes: FALSE POSITIVE, NO ACTION. The premise -- that ticket titles carry a full relative path -- is wrong: `dev_review.create_tickets` builds every ticket title from `Path(r["file"]).name` (dev_review.py:236), i.e. the BASENAME, so the fingerprint's file field and `_is_dismissed`'s `Path(file_path).name` comparison are both basenames by construction (this very ticket's own heading, "[_review_gate.py] ...", demonstrates it). Verified end-to-end against the REAL ticket corpus: `load_dismissed_fingerprints("Tickets")` yields 8 basename-keyed fingerprints, and replaying TICKET-0179's exact finding title against `orchestrator/image_audit.py` through `decide()` returns PASS with suppressed=1, blocking=0 -- the suppression the reviewer claims can never fire, firing. (The reviewer's suggested "fix" -- comparing the finding's full relative path against the fingerprint -- would BREAK matching, since fingerprints are basenames.) Known accepted limitation, safe direction: hand-written multi-file headings like "[sequencer.py/nodes.py]" never match a basename, so those tickets simply don't suppress (no suppression = the gate still blocks; never the reverse). No code change made.
