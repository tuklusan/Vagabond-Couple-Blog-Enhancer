# TICKET-0194: [vision_client.py] Format string vulnerability in CERTIFY_PROMPT
Status: Closed
Priority: High
Type: Bug
Created: 2026-07-05
Description: The _CERTIFY_PROMPT is used with string interpolation (%) with user-controlled data (original and proposed). If either contains format specifiers (e.g., '%s', '%d'), it will crash or produce incorrect formatting, and could potentially be exploited. This is a security risk because an attacker controlling the alt text or caption could cause arbitrary format string attacks leading to information disclosure or denial of service. | Suggestion: Replace % formatting with .format() or f-strings using explicit placeholders to avoid interpretation of user data as format specifiers. For example: _CERTIFY_PROMPT.format(field=field, field_upper=field.upper(), original=original or '(none)', proposed=proposed or '') and modify the prompt string to use named placeholders {field}, {original}, etc. | File: orchestrator/vision_client.py | Severity: critical
Steps to Reproduce: 
Notes: DUPLICATE of TICKET-0189 (closed FALSE POSITIVE the same day, worded differently enough -- 'Format string vulnerability' vs 'Unsafe string formatting' -- that this batch's fingerprint match (0.25 token overlap) fell short of the 0.6 suppression threshold; a known, accepted limitation of TICKET-0183's token-based matching). Same refutation applies verbatim: '%' in a %-template is only interpreted in the TEMPLATE, never in substituted values -- verified again here by re-running the adversarial-value render from 0189 (values containing %, %s, %(x)s, and {} all pass through verbatim). The suggested .format()/f-string rewrite would introduce brace-injection on any caption containing literal '{' or '}', which the current % scheme does not have. No code change made.
