Description: test_derived_waypoints_string_type_ignored patches context_extractor.derive_route_from_prose directly. If extract_context or other modules use a local reference (e.g., `from .context_extractor import derive_route_from_prose`), the patch will not propagate, making the test ineffective. Notes: FALSE POSITIVE, NO ACTION. The premise does not apply to this call pattern: `extract_context` invokes `derive_route_from_prose(html)` as a bare module-global NAME within its own module, and Python resolves module-global names at CALL time against the module's __dict__ -- so assigning `context_extractor.derive_route_from_prose = <mock>` before the call is fully effective ('import-time binding' only bites when a caller does `from mod import func` and calls the LOCAL reference, which is not the case here). Empirical proof already in the suite: test_derived_waypoints_string_type_ignored's assertions can only pass if the mock runs -- the mock's sentinel values ('Foo, Bar, Baz' string / ['Foo','Bar'] list) are asserted in the RESULT, and with allow_llm=True an unpatched path would attempt a real provider call and return different data (or fail). Both directions of the test (string ignored, list applied) pass deterministically. The patch is also restored in a finally block, so no cross-test leakage. No code change made.
