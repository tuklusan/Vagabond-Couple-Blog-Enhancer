# TICKET-0180: [vision_client.py] Path traversal in Wikimedia URL construction allows SSRF agai
Status: Closed
Priority: High
Type: Bug
Created: 2026-07-04
Description: The function downscaled_url builds a thumbnail URL by directly concatenating captured groups from the original URL. An attacker can craft a src like https://upload.wikimedia.org/wikipedia/commons/a/ab/../../../sensitive/path to traverse outside the thumbnail directory on the same host. This can bypass intended fetch restrictions and potentially expose internal resources or sensitive files, constituting a Server-Side Request Forgery (SSRF) vulnerability. | Suggestion: Sanitize the file-name group by URL-decoding, eliminating path traversal sequences (../), and re-encoding before insertion. Alternatively, enforce that the final URL stays under the /thumb/ prefix by parsing the path and validating its resolution. | File: orchestrator/vision_client.py | Severity: critical
Steps to Reproduce: 
Notes: FALSE POSITIVE, NO ACTION. Two independent reasons: (1) the filename capture group is `[^/]+` -- a literal '../' contains slashes, so such URLs simply do not match the Wikimedia pattern and pass through unchanged to the host-validating fetcher (verified: the example URL from this ticket returns unmatched/unchanged). (2) Even for a percent-encoded traversal that does match (no literal slash), the constructed URL's HOST is invariantly upload.wikimedia.org -- a fixed public host; requests sends the path un-decoded, the server treats it as a literal filename and 404s, which fetch_image already handles as an ordinary failure. A request whose target host cannot be influenced is not SSRF; there are no 'internal resources' reachable by path manipulation on a public CDN host. All fetches additionally go through safe_get (TICKET-0178), which re-validates every redirect hop. No code change made.
