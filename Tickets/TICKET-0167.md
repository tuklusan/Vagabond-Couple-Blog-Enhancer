# TICKET-0167: [vision_client.py/image_audit.py] Phase 1/1J visual image audit -- detect and correct image-vs-metadata/caption mismatches via NIM vision models
Status: Closed
Priority: Medium
Type: Feature
Created: 2026-07-04
Description: User-reported (alaska1 test case, Vancouver-departure section): some photographs visibly disagree with their alt/title/caption text -- e.g. a caption claiming "MS Statendam upper decks during departure" over a frame that actually shows Canada Place's sail roofline, and an "aircraft cabin during daylight flight" alt over a photo of an airport terminal. No text-only pass can ever catch this class of defect; the pipeline needed a pass that actually LOOKS at each image. User directive: "add a thorough visual image-inspection pass for meta-data+caption accuracy review and correction" using the NVIDIA NIM API key already in the environment.
Steps to Reproduce: See alaska cruise original, images 1 (terminal vs "aircraft cabin"), 11 (Canada Place vs "ship upper decks"), 12 (harbor view vs "three-story atrium with Fountain of the Siren").
Notes: Implemented as Phase 1 / 1J (1A-1I already taken; "1E" is the YouTube re-emit):
- `orchestrator/vision_client.py` (new): NIM vision chain meta/llama-3.2-90b-vision-instruct -> nvidia/nemotron-nano-12b-v2-vl -> microsoft/phi-4-multimodal-instruct (all verified available on this account via /v1/models; primary overridable via ORCH_VLM_MODEL). Images fetched at a downscaled size via Blogger URL size-token rewrite to `/s512-rj/` (-rj forces server-side JPEG re-encode -- verified live: a 546KB PNG comes back as an 87KB JPEG, fitting NIM's ~180KB inline-base64 ceiling), SSRF-guarded by the TICKET-0159 helper, one smaller-size retry (`/s320-rj/`), hard byte cap. Unparseable/markdown/truncated verdicts fall through to the next model (observed live and fixed: models occasionally answer in markdown despite JSON-only instructions).
- `orchestrator/image_audit.py` (new): per-image verdict MATCH / PLAUSIBLE / CONTRADICTED for each of alt/title/caption. Anti-hallucination policy: proper nouns, dates, coordinates, VANTAGE POINTS, and details beyond the frame are unverifiable from pixels -> always PLAUSIBLE; CONTRADICTED is reserved for a wrong PRIMARY SUBJECT (text says totem pole, frame shows a dining room). Corrections must keep the original's place names/journey context unless the location itself is the contradiction. Deterministic gates before any correction is applied: writing-rules clean (forbidden words/narrator), sane length, linked captions never auto-edited (G3; finding-only), and caption corrections must retain at least one proper token from the original (observed live: the model occasionally "corrects" a caption down to a bare visual description, silently deleting "Vancouver" -- an SEO/information regression rejected deterministically). Per-image verdicts cached incrementally in the `1J_image_audit_progress` artifact (G4-resumable; a 424-image audit that dies mid-run resumes where it stopped). All VLM prompts/responses logged to ai_transcript.txt.
- `orchestrator/sequencer.py`: new `image_audit_node()` after 1I, before the Phase 4 gate (audit counts appear in the Phase 4 approval summary); audit outage/missing key records status=unavailable and CONTINUES (the audit improves the post; it must never block the pipeline). ORCH_IMAGE_AUDIT=0 disables; ORCH_IMAGE_AUDIT_LIMIT caps image count.
- `orchestrator/assembler.py`: `apply_image_corrections(html, corrections)` -- deterministic application of gated corrections (alt/title attributes; caption text only re-verified link-free at apply time), wired into `assemble()` / `_assemble_working()` as one more Phase 5 transform.
- `tests/test_image_audit.py` (new, VLM mocked): 27 checks -- record collection, URL rewrite forms, SSRF rejection, contradiction gating, rule-breaking/context-dropping correction rejection, linked-caption refusal at both audit and apply time, unknown-status degradation to PLAUSIBLE, application correctness.
- Live verification (15 alaska images, Vancouver departure section): 15/15 audited with 0 fetch and 0 verdict failures after fixes; genuine mismatches caught and corrected with location retained (terminal-vs-cabin, harbor-vs-atrium); place-dropping corrections correctly demoted to finding-only; prior over-flags (poetic detail beyond the frame; vantage-point claims) eliminated by the tightened prompt. Residual run-to-run variance in which borderline images get flagged is inherent to VLM judgment and acceptable: worst case is a finding-only record for the operator.
