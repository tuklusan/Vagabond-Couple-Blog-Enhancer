Status: Closed

Description: Multiple tests create RunState with hardcoded run IDs (e.g., 'test_resume', 'test_block') and rely on disk-persisted state. The test_resume test calls RunState.load after create, assuming earlier run persisted. No cleanup is performed, so leftover state from previous runs can cause false failures. This makes the suite non-deterministic and fragile in CI or repeated local runs. | Suggestion: Use unique, temporary run IDs (e.g., with a random suffix) for each test run, and ensure any persisted state directories are cleaned up before and after tests. Alternatively, mock the persistence layer to avoid disk I/O. | File: tests/test_sequencer.py | Severity: critical

The premise -- 'leftover state from previous runs can cause false failures' -- is refuted by RunState.create's own documented behavior: reusing a run_id deliberately REINITIALISES it (state.py create(): overwrites status.json, artifacts, working.html; prints an explicit warning), so a prior run's leftover state can never leak into a fresh test invocation -- that IS the isolation mechanism, not an absence of one. test_resume_skips_completed_nodes' RunState.load call reads state created earlier IN THE SAME TEST FUNCTION during THE SAME process, which is the resume feature under test (TICKET-0174), not cross-run pollution. Verified empirically: tests/test_sequencer.py passes identically across 3 consecutive fresh runs. This project also runs tests directly and sequentially (`python tests/test_X.py`, no pytest/xdist/parallelism), so no concurrent-access race exists either. No code change made.
