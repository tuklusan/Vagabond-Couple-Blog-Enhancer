# TICKET-0152: Phase 5 no_forbidden gate hard-fails on words already present in the untouched original prose
Status: closed
Priority: High
Type: Bug
Created: 2026-07-03
Description: document_deterministic_checklist's no_forbidden check re-scans the WHOLE assembled document for banned AI-cliche words/phrases, with no distinction between a word the orchestrator's own generated content introduced vs. a word that was already present in the ORIGINAL author's untouched prose. Phase 1I ('Writing-rules audit -- existing prose') already runs this exact scan read-only/advisory on the source before any generation and reports it (clean=False, forbidden=N) without halting -- but Phase 5 then hard-blocks on the SAME pre-existing occurrences. On a short post this rarely matters (little original prose to trip the list); on a long, prose-rich original (observed: alaska-cruise post, ~900KB, 424 photos) genuine human writing incidentally contains words like 'naturally', 'indeed', 'testament to', 'leverage' in completely ordinary, non-AI-cliche usage (e.g. describing bird-wing leverage), making Phase 5 permanently uncertifiable for content the orchestrator was never asked or able to touch. FIX: added _new_forbidden_hits(html, baseline_forbidden) in review_loop.py -- only forbidden-term hits whose COUNT increased relative to the Phase 1I baseline (read from the analysis_1I_writing_rules artifact) now block Phase 5; threaded baseline_forbidden through document_deterministic_checklist()/run_document_certification(). A generated fragment that introduces or repeats a banned term still fails; a word already sitting unchanged in 50-year-old human prose does not. Verify: re-run the alaska-cruise post and confirm Phase 5 no longer blocks on pre-existing forbidden words.
